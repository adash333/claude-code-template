### アシスタント（要約）

- 影響を明示して確認を取った:
  - すべての中間履歴が失われる
  - リモートに対して **force push** が必要
  - clone/fork している人がいると衝突する
- 提示した方式:
  - **A. orphan ブランチ方式**（推奨）: `git checkout --orphan` で親なしブランチを作成 → 全 add → commit → 旧 main 削除 → リネーム → force push

- ユーザー回答: 「a」（A 方式・コミットメッセージは `Initial commit`）
- 実行:

  ```bash
  git checkout --orphan new-main
  git add -A
  git commit -m "Initial commit"
  git branch -D main
  git branch -m new-main main
  git push -f origin main
  ```

- 結果: 単一コミット `7f47d84 Initial commit`（38 files, +7506）。リモートも `d455fa7...7f47d84 (forced update)` で上書き完了

## やり取り 15: Skill と Command の役割整理 → Skill に集約

### ユーザー

> あれ？スキルとコマンドと何が異なるのですか？今回は、`/save-prompt-log` は Skill が妥当ですか？

### アシスタント（要約）

Skill と Command の違いを整理して回答:

| | Skill | Slash Command |
| --- | --- | --- |
| 配置 | `.claude/skills/<name>/SKILL.md` | `.claude/commands/<name>.md` |
| 起動方法 | ① `/<name>` で明示起動 / ② Claude が文脈から自動起動 / ③ Skill ツール経由 | `/<name>` で明示起動のみ |

**重要な発見**: Claude Code では `/<name>` で **Skill も Command も両方とも起動できる**。よって単機能の Skill に対して同名の Command ファイルを置くと冗長で、しかも同名重複によりどちらが起動するか曖昧になる。

このセッション初頭、まだ `/save-prompt-log` の Command ファイルを作る前に、ユーザーは既に `/save-prompt-log` で Skill を直接起動できていた事実が、この冗長さの裏付け。

### 決定事項（Skill / Command の整理ルール）

このプロジェクト内での命名・配置ルール:

1. **「単機能の Skill を呼ぶだけ」のもの** → **Skill のみ**（Command は作らない）
   - 該当: `save-prompt-log`, `draft-idea`
2. **「複数の Skill を順序立てて呼ぶワークフロー」** → **Command として定義**
   - 該当: `setup-project`（PRD → 機能設計 → アーキテクチャ → ... の 6 Skill を順番に呼ぶ）
3. **「Skill を持たない独自ロジック」** → **Command のみ**
   - 該当: `add-feature`, `review-docs`

### ユーザー回答

> はい、Skill に集約してください

### 実施した整理

**削除した冗長ファイル:**

- `.claude/commands/save-prompt-log.md`
- `.claude/commands/draft-idea.md`

**残った Command（3つ）:** `setup-project`, `review-docs`, `add-feature`

**README 更新:**

- スラッシュコマンド表: 該当 2 行を削除し、「Command として定義する基準」を 1 文で明示
- Skill 表: 「`/<skill-name>` で直接起動できる」旨を 1 文追加し、`draft-idea` / `save-prompt-log` の説明を詳細版に差し替え

ユーザー体験は不変（`/draft-idea` も `/save-prompt-log` もそのまま使える）、ファイル構成と README は重複が解消されシンプルに。
