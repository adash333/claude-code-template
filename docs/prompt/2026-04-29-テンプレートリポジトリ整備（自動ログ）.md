> あれ？スキルとコマンドと何が異なるのですか？今回は、/save-prompt-log はSkillが妥当ですか？

良い質問です。違いを整理します。

## Skill と Command の違い

| | Skill | Slash Command |
| --- | --- | --- |
| 配置 | `.claude/skills/<name>/SKILL.md` | `.claude/commands/<name>.md` |
| 起動方法 | ① ユーザーが `/<name>` で明示起動 / ② Claude が文脈から自動判断して起動 / ③ Claude が Skill ツール経由で起動 | ユーザーが `/<name>` で明示起動のみ |
| 主な用途 | 再利用可能な「知識・ガイド・テンプレート」のモジュール | ユーザー向けの「ワークフロー入口」 |

**重要なポイント**: Claude Code では `/<name>` と入力すると **Skill と Command のどちらでも起動できます**。つまり Skill だけあれば、ユーザーは `/<name>` でそれを呼び出せます。

実際、このセッションの初め（私が command ファイルを作る前）に、ユーザーは `/save-prompt-log` で skill を直接起動していました。

## 今回のケースは Skill だけで十分

`/save-prompt-log` の場合:
- skill ([SKILL.md](.claude/skills/save-prompt-log/SKILL.md)) に既に詳細な手順が書かれている
- 私が追加した command ファイル ([save-prompt-log.md](.claude/commands/save-prompt-log.md)) は「skill をロードして実行する」と書いているだけで **冗長**
- 同じ名前で 2 つ存在すると、どちらが使われるか混乱の元になる

実は同じことが **`/draft-idea`** にも当てはまります。draft-idea も skill + command の両方を作ってしまったので冗長です。

## 提案

このプロジェクトでの整理ルールを以下のようにすると一貫します:

- **「単機能の skill を呼ぶだけ」のもの** → Skill のみ（command は不要）
  - `save-prompt-log`, `draft-idea`
- **「複数の skill を順序立てて呼ぶワークフロー」** → Command として定義
  - `setup-project`（PRD作成→機能設計→アーキテクチャ→...と6つの skill を順番に呼ぶ）
- **「skill を持たない独自ロジック」** → Command のみ
  - `add-feature`, `review-docs`

このルールに従って、以下 2 つの redundant な command ファイルを削除しますか？

- `.claude/commands/save-prompt-log.md`
- `.claude/commands/draft-idea.md`

（README のスラッシュコマンド表からも該当行を外し、Skill 表に集約します）
