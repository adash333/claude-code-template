# Draft Idea スキル

ユーザーに段階的な質問を投げかけ、その回答をもとに `docs/ideas/initial-requirements.md` を生成するスキルです。テンプレートの `{...}` プレースホルダーを手作業で埋める代わりに、対話形式で必要な情報を引き出します。

## 出力先

```text
docs/ideas/initial-requirements.md
```

## 参考ファイル

- テンプレート構造: [docs/ideas/initial-requirements.md](../../../docs/ideas/initial-requirements.md)（プレースホルダー版）
- 記入例: [docs/ideas/initial-requirements.example.md](../../../docs/ideas/initial-requirements.example.md)（TaskCLI のサンプル）

最終出力はテンプレートと同じセクション構造に従ってください。

## 実行手順

### 手順 0: 既存ファイルの状態を確認

1. `docs/ideas/initial-requirements.md` を Read する
2. 内容を判定:
   - **テンプレート状態**（`{プロダクト名}` などのプレースホルダーが残っている）→ そのまま上書きする旨を 1 行で告げて Q&A へ
   - **既に編集済み**（プレースホルダーが消えており、実際の内容が書かれている）→ ユーザーに確認:
     - 「既存内容を上書き」
     - 「キャンセル」
     - 「別ファイル名で保存（例: `initial-requirements.draft.md`）」
   - **ファイルが存在しない** → そのまま Q&A へ

### 手順 1: Q&A の進め方

質問は **セクション単位でまとめて** 投げるが、1 度にすべてを聞かない。**Phase 1 → Phase 2 → ...** と段階的に進め、各 Phase の回答が揃ってから次に進む。

各質問は以下の原則で:

- 例を 1 つ添える（記入例ファイルから抜粋して構わない）
- 「わからない」「スキップ」と回答された場合はそのセクションを未入力のままにする（テンプレートのプレースホルダーをそのまま残す or `（未定）` と書く）
- ユーザーが端的に答えた場合は、Claude 側で自然な文章に整える（要約しすぎない）

### 手順 2: Q&A フェーズ

#### Phase 1（必須）: プロダクトの核

> 以下を教えてください:
>
> 1. **プロダクト名**（仮でもOK）
> 2. **一言で言うと**（30文字以内のキャッチコピー）
> 3. **誰のために、何を解決するプロダクトか**（2-3文）

#### Phase 2（必須）: 解決したい課題

> 今 困っていること・解決したい課題を **2〜3個** 挙げてください。各課題について:
>
> - 課題のタイトル（短く）
> - 具体的な状況・場面
> - なぜそれが問題なのか
>
> 思いつかない場合は「主な課題は1つだけです」「まだ整理できていません」でも構いません。

#### Phase 3（必須）: ターゲットユーザー

> メインのユーザー像を 1〜2 人分、ペルソナとして教えてください。各ペルソナについて:
>
> - 呼び名（例: 「個人開発者」「小規模チームのリーダー」）
> - 職業・年齢層・経験
> - そのペルソナ特有の悩み（2-3個）
> - そのペルソナが欲しいもの（2-3個）

#### Phase 4（必須）: MVP の機能候補

> MVP（最小限のプロダクト）に含めたい機能を **P0（絶対必要）** として挙げてください。各機能について:
>
> - 機能名
> - 1-2 行の説明
> - 可能ならコマンド例・UI 例
>
> 加えて、**P1（重要だが後回しでよい）**・**P2（あったら嬉しい）** に分けて、思いつくものがあれば挙げてください（任意）。

#### Phase 5（任意）: 差別化ポイント

> 既存の類似プロダクトと比べた強みは何ですか？スキップしても構いません。

#### Phase 6（任意）: 技術的な検討事項

> 採用予定 / 検討中の技術スタックがあれば教えてください（言語・フレームワーク・データ保存方法など）。スキップ可。

#### Phase 7（任意）: 非機能要件・成功指標

> 性能目標・ユーザー体験の目標・成功指標などのざっくりした想定があれば教えてください。スキップ可。

### 手順 3: ドラフト生成

すべての回答が揃ったら、テンプレートの構造に従って `docs/ideas/initial-requirements.md` を生成します。

**書き込みのルール:**

1. テンプレートの全セクションを残す（任意フェーズで未回答のセクションも、プレースホルダーまたは `（未定 / 後で記入）` として残す）
2. プレースホルダー `{...}` は回答内容に置き換える
3. テンプレート冒頭の「テンプレート使用方法」ブロック（`> **テンプレート使用方法**:` で始まる引用）は **削除する**（生成済みファイルでは不要）
4. テンプレート末尾の「次のステップ」セクションはそのまま残す
5. ヒント用 `<!-- ... -->` コメントは残す（後から見直すときの参考になる）

### 手順 4: ユーザー確認とレビュー

書き込み後、以下を 1 メッセージで報告:

```text
✅ docs/ideas/initial-requirements.md を作成しました。

含めた内容:
- プロダクト名: {名前}
- 課題: {N}件
- ペルソナ: {N}人
- MVP機能（P0）: {N}件
{任意セクションは「未入力」または含まれた旨}

次のステップ:
- ファイルを開いて確認・微修正してください
- 完了したら `/setup-project` で永続ドキュメントを作成できます
```

## 注意事項

- **質問を一度に投げすぎない**: Phase 単位で区切る。1 つのメッセージで 7 Phase 全部聞かない
- **既存ファイルを勝手に上書きしない**: 編集済みファイルがあるときは必ず確認
- **テンプレートの構造を維持**: 勝手にセクションを消したり順番を入れ替えたりしない
- **回答を脚色しすぎない**: ユーザーが端的に答えたものを過剰に膨らませない
- **「わからない」を尊重**: 任意セクションでスキップされたら無理に埋めようとしない

## 完了条件

- `docs/ideas/initial-requirements.md` が存在し、Phase 1〜4（必須）の内容が反映されている
- ユーザーに完了報告と次のアクション（`/setup-project`）を案内している
